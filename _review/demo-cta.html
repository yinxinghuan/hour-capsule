Hour Capsule — Demo Mode 评审

分享链接被在外部浏览器（非 AlterU iframe）打开时的降级页面。 判定条件：URL 缺少 api_origin + telegram_id 这两个 query param → isInAigram === false。 Field tab 替换成 FieldDemoCTA 组件。 核心玩法仍可用（gen-image + LLM 都是 CORS * 直 fetch，已实测）： Collect 按钮真出袋、Reveal 动画真播、Detail 真开、Altar tab 真存 localStorage。 只有 Field 跨用户墙 和 profile 头像 在外部废掉。

STATE 1

First-time visitor — 还没装过袋

分享链接的最常见落点：一个完全不知道 AlterU 是什么的人点进来。 没有 hero 胶囊（mirror.capsules 为空）就把游戏官方海报（pocket watch 真空袋）顶上来当"示例袋", 让来访者 先看到游戏的产出形态再决定是否点 Collect。 EXAMPLE tag 用粉色 + 边框, 跟 "YOUR LATEST" cream tag 区分开, 避免"我什么时候装的这个"的迷惑。

UX 决策

还在桌上的几个问题

1. Detail 页的 Like 按钮 — Demo 模式下 like count 永远 0（没有跨用户数据）。 保留 Like 按钮 = "看起来像正常游戏" 但 count 始终空；隐藏 = 更诚实, 少一个无效控件。 当前实现: 保留。

2. CTA 的位置 — 现在 CTA 在 hero + pitch 之后 (中段)。是否应该 悬浮在底部 永远可见 (类似 sticky bottom CTA)？争议点: TabBar 已占底部, sticky CTA 会和它打架; 中段 CTA 又不一定每个 viewport 都看见。 当前实现: 中段。

3. Phantom counter — "427 others sealed today" 是按小时 seed 的假数字。 是否换成真数字? 可以加一个无认证的 /aigram/api/today-count 接口暴露聚合统计。或者干脆不写数字。 当前实现: 假数字, 缓慢累加。

4. 多次 Collect 后的体验 — 没 cap, 同一浏览器每小时可以无限收。 是否应该收 3 个后 hard-gate "下载 app 才能继续"? 反对: 用户体验越好越想下载; 支持: 强制下载也算转化。 当前实现: 无 cap, 跟正常一样的 1/hour 冷却。

5. Aigram 旧字眼 — 代码 / 文件名仍是 isInAigram / api_origin / callAigramAPI / postAigramAPI / openAigramProfile。 这是平台 runtime 接口的内部命名 (历史), 不影响用户。是否要重命名 (大改) 或就留着 (注释说明)? 当前实现: 留着, FieldDemoCTA 注释里有说明。
